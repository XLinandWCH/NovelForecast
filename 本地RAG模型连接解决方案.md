# RAG 本地模型连接问题解决方案

## 1. 问题现象
当在应用中使用 Nexa AI 运行本地 Embedding 模型（如 `embeddinggemma-300m-npu`）时，系统报错无法连接或 RAG 失败。
控制台日志显示：`OPTIONS /v1/embeddings 404`。

## 2. 根本原因分析
1.  **CORS 跨域限制**：浏览器出于安全考虑，在向非同源地址（从 `localhost:5173` 到 `127.0.0.1:18181`）发送复杂请求（如 POST + JSON）前，会先发送一个 `OPTIONS` 预检请求。
2.  **服务端不支持预检**：本地运行的 Nexa AI (GIN 框架) 默认并未配置处理 `/v1/embeddings` 的 `OPTIONS` 方法，导致返回 404，浏览器随即拦截了后续真正的 embedding 请求。
3.  **内网穿透与公网访问**：如果应用通过公网或内网穿透（如 cpolar, ngrok）访问，浏览器无法直接解析并连接客户端电脑上的 `127.0.0.1`，必须通过应用服务器进行代理中转。

## 3. 实施的修复方案
我们采用了 **Vite 级联代理（Reverse Proxy）** 技术解决了此问题：

### 修改 A：配置开发服务器代理 (`vite.config.ts`)
我们在 Vite 配置中添加了 `/api/rag` 路由代理：
- **拦截预检**：配置代理服务器在接收到 `OPTIONS` 请求时，跳过转发，由代理节点直接返回 `204 No Content` 响应及跨域头，彻底骗过浏览器安全检查。
- **转发请求**：将 `/api/rag` 后的所有内容透传给本地端口 `18181`。

### 修改 B：智能路由选择 (`src/utils/ragClient.ts`)
修改了 RAG 客户端的请求逻辑：
- **动态识别**：代码会自动检查配置的 `Base URL` 是否包含 `127.0.0.1` 或 `localhost`。
- **重定向**：如果是本地模型，请求不再发往原地址，而是发往前端自身的 `/api/rag` 路径。这样，浏览器认为请求是在“同一个服务器”内，无需 CORS 检查，剩下的交给 Vite 后端与 Nexa AI 通信。

## 4. 为什么不建议直接在代码里写死内网穿透地址？
用户提到“部署上线后内网穿透会失效”，这也正是我们采用 Vite 代理的原因：
- **解耦环境**：代理配置允许你在部署时通过环境变量（如 `VITE_RAG_PROXY_TARGET`）轻松修改后端地址，而无需修改核心业务代码。
- **稳定性**：代理模式在生产环境下（配合 Nginx 或相似服务）能提供更强的安全过滤、负载均衡及错误处理机制，比浏览器直连要稳定得多。

## 5. 操作建议
1.  **确保 Nexa AI 服务已启动**：
    ```bash
    nexa serve NexaAI/embeddinggemma-300m-npu
    ```
2.  **设置界面配置**：
    - 提供商选择 `Nexa AI`。
    - API Base URL 保持默认或填写 `http://127.0.0.1:18181/v1`。
3.  **刷新页面**：修改配置后，请刷新浏览器以应用最新的代理路由。
